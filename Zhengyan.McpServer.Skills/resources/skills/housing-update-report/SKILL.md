# 住房维度更新报告格式化 Skill

用于将住房维度更新内容整理成正式、完整、适合直接提交的 Markdown 报告。只适用于住房维度材料；如果任务是全市或多维度的城市体检宏观更新建议，应改用 `cityagent-macro-update-report`。

## Use When

- 用户明确要求把原始内容“整理成报告”“改写成正式材料”“输出 Markdown 报告”
- 输入内容与住房维度更新、住房问题、更新建议、立项依据、项目建议有关
- 需要把零散内容统一改写成固定章节结构
- 任务明确只针对住房维度，而不是“住房 + 小区 + 街区 + 城区”的城市级综合报告

## Required Output

- 最终只返回 Markdown 报告
- 不要返回 JSON
- 不要把最终结果包在 Markdown 代码块里
- 必须完整覆盖报告模板中的所有一级和二级章节

## Workflow

1. 先通读用户提供的全部 `content`，识别可归入以下四类的信息：
   - 住房问题
   - 更新建议
   - 立项依据
   - 项目建议
2. 再通过 `ReadSkillFile(skillId="housing-update-report", relativePath="references/report-template.md")` 读取模板，严格按模板组织章节。
3. 将输入内容改写为正式、简洁、适合汇报材料的表述：
   - 保留用户已经提供的核心事实
   - 不照搬口语化、碎片化或清单式原文
   - 对标题进行概括提炼，但不要夸大
4. 若任务中提供了 `text.txt`：
   - 只借鉴其章节组织、语气和表达颗粒度
   - 不得复用其中的地名、数字、年份、政策名称、项目名称
5. 当信息不足时：
   - 允许使用稳妥概括
   - 不得擅自补充具体统计值、政策编号、项目投资额、责任单位名称
   - 可使用“待进一步明确”“可在后续论证中补充完善”这类保守表述

## Hard Rules

- 只能基于本次输入内容生成报告
- 不得虚构输入中没有提供的核心事实数据
- 如果任务是城市级、多维度、宏观更新建议，不要继续使用本 Skill，改走 `cityagent-macro-update-report`
- 若原始输入未给出具体政策名称，只能概括写“具备相关政策支持基础”之类的稳妥表述
- 若原始输入未给出明确资金来源，只能概括写“资金安排可结合后续统筹渠道进一步研究落实”
- 若原始输入未给出明确责任单位或投资测算，只能写为后续深化阶段进一步明确
- 风格保持正式、简洁、完整，避免口号式和宣传式语言

## Writing Notes

- “住房问题”与“更新建议”优先使用编号列表
- “立项依据”必须保留“政策支持”“资金支持”两个二级小节
- “项目建议”应尽量覆盖项目目标、建设内容、实施方向、责任分工、投资测算等；没有信息时用保守表述补足结构，但不要补造事实
- 如输入内容只够支撑 1 个问题或 1 条建议，也仍然保留完整章节结构，可将同类内容整合后表述为“重点问题”“重点建议”
