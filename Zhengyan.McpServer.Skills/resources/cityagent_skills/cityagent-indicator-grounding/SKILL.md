# ⚠️ 指标编码归一化 Skill（输出规范：不要出现"类型1"、"住房（类型1）"等字眼）

用于把用户说的指标名、问题类型、口语化描述，映射成可靠的 `indicator_code`，避免把自然语言直接传给需要编码的工具。

## Use When

- 需要调用 `list_indicator_targets`
- 用户问“某类问题对应哪些小区、点位或对象”
- 用户给的是自然语言问题描述，而不是正式指标名
- 你需要判断某个候选指标属于住房、小区、街区还是城区维度

## Mandatory Query

始终先调用：

`query_data(query="<用户原始指标名或问题描述>", sourceFiles="问题指标表.db", topN=5)`

必要时可做第二次召回，但第二次必须是更正式的改写，不能凭空捏造编码。

## 问题类型编码（重要基础）

在调用需要问题类型参数的接口时，必须使用以下编码：

| 编码 | 问题类型 | 说明 |
|------|----------|------|
| 1 | 住房 | 住宅楼栋、住房单元等 |
| 2 | 小区（社区） | 小区、新村、花园、宿舍等 |
| 3 | 街区 | 街道、片区 |
| 4 | 城区（城市） | 城区、城市级 |

**重要：内部编码仅用于接口调用，在回答用户时必须转换为对应的中文名称！**

## 输出格式规范（重要！必须遵守）

当接口返回数据中包含 `dimension_type`、`target_type`、`issuesType` 等类型字段时，**必须**将编码转换为用户可读的中文名称：

| 接口返回 | 回答用户时必须说 |
|----------|-----------------|
| `"1"` | 住房 |
| `"2"` | 小区（社区） |
| `"3"` | 街区 |
| `"4"` | 城区（城市） |

**禁止在回答中出现以下内容：**
- ❌ "类型1"、"类型2"、"类型3"、"类型4"
- ❌ "维度1"、"维度2"、"维度3"、"维度4"
- ❌ "住房（类型1）"、"小区（类型2）"、"街区（类型3）"
- ❌ 任何数字编码形式

**🚨 特别注意：不要在括号内显示编码！**
- ✅ "住房问题184条" 或 "住房：184条"
- ❌ "住房（类型1）问题184条" ← 这是错误的！
- ❌ "住房类（类型1）共184条" ← 这是错误的！
- 在表格或列表中，只写中文名称，不要加括号说明类型

**正确示例：**
- ✅ "住房问题共184条"
- ✅ "小区（社区）问题数量为73条"
- ✅ "街区问题有28条"

**错误示例（绝对禁止）：**
- ❌ "住房（类型1）问题184条"
- ❌ "住房类（类型1）共184条"
- ❌ "小区（类型2）73条"

## Fields to Trust

- `indicator_name`：正式指标名称
- `indicator_code`：后续工具要用的真实编码
- `first_dimension`：中文维度名
- `dimension_type`：维度编码
  - `1 = 住房`
  - `2 = 小区（社区）`
  - `3 = 街区`
  - `4 = 城区`

## Selection Rules

1. 先判断用户问的是住房、小区、街区还是城区层面，再选对应 `dimension_type` 的候选。
2. 如果用户已经限定“小区”“社区”“某个小区有哪些这类问题”，优先看 `dimension_type=2` 的候选。
3. 如果用户问的是城市级或区级指标表现，优先看 `dimension_type=4` 的候选。
4. 如果用户给的是非常口语化的现象词，先看召回结果是否真的同主题；不要因为有返回结果就强行采用。
5. 可把 `_similarity >= 0.55` 视为较强信号；当 top1 `< 0.50` 或前几条明显跨主题时，视为未可靠命中。

## Two-Pass Strategy

- 第一轮：直接用用户原话召回。
- 第二轮：若结果弱或歧义大，把原话改写成更正式的指标表达后再召回一次。
- 只有两轮结果都指向同一指标或同一主题时，才放心下游调用。

## Example

- 用户问：“需要智慧化改造的小区有哪些？”
- 先调 `query_data(query="智慧化改造小区", sourceFiles="问题指标表.db")`
- 若命中 `indicator_name = 需要进行智慧化改造的小区数量（个）`, `indicator_code = N-SRC`, `dimension_type = 2`
- 再调用 `list_indicator_targets(indicator_code="N-SRC", area_code?)`

## Bad Example

- 用户只说：“占道经营”
- 如果召回前几条是 `绿道服务半径覆盖率`、`乱停乱放车辆问题的道路数量` 这类明显不同主题的结果，说明当前召回不可靠。
- 这时不要直接选 top1，而要改写查询或请用户补充更正式的指标描述。

## Hard Rules

- 不要自己编 `indicator_code`。
- 不要忽略 `dimension_type`，否则很容易把城区指标拿去做小区级查询。
- 当 `indicator_name` 看起来和用户问题不是一回事时，宁可停下来澄清，也不要继续调用。

## 指标名称规范（重要）

- **必须使用完整指标名称**进行回答和展示，如"存在结构安全隐患的住宅数量（栋）"、"中学服务半径覆盖率（%）"等
- **禁止使用指标代码**如"N-SSHU"、"N-GSHU"等，也**禁止使用"指标1xxx"、"维度1xxx"这类代号式称呼**
- 从 `query_data` 召回结果中获取 `indicator_name` 字段作为标准名称使用
- 在回答用户时，只展示指标名称，不展示指标代码
- 如果后端返回的数据只有代码没有名称，必须先通过 `query_data(..., sourceFiles="问题指标表.db")` 查询补全名称后再回答
