# ⚠️ 城市体检宏观更新建议报告 Skill（输出规范：不要出现"类型1"、"住房（类型1）"等字眼）

用于把城市体检的多维度问题、更新建议、立项依据和项目建议整理成正式 Markdown 报告，适用于“城市体检宏观更新建议”“城市更新正式材料”“城市问题 + 更新建议 + 立项依据 + 项目建议汇报稿”等场景。

## Use When

- 用户明确要求"城市体检宏观更新建议""城市问题 / 更新建议 / 立项依据 / 项目建议"
- 输入内容涉及住房、小区（社区）、街区、城区（城市）至少两个维度
- 需要输出正式 Markdown 报告，而不是自由分析或简短总结
- 如果任务只针对住房维度，不用本 Skill，改用 `housing-update-report`

## 问题类型编码（重要基础）

在调用需要问题类型参数的接口时，必须使用以下编码：

| 编码 | 问题类型 | 说明 |
|------|----------|------|
| 1 | 住房 | 住宅楼栋、住房单元等 |
| 2 | 小区（社区） | 小区、新村、花园、宿舍等 |
| 3 | 街区 | 街道、片区 |
| 4 | 城区（城市） | 城区、城市级 |

**重要：内部编码仅用于接口调用，在回答用户时必须转换为对应的中文名称！**

## 输出格式规范（重要！必须遵守）

当接口返回数据中包含 `target_type`、`issuesType` 等类型字段时，**必须**将编码转换为用户可读的中文名称：

| 接口返回 | 回答用户时必须说 |
|----------|-----------------|
| `issuesType: "1"` | 住房 |
| `issuesType: "2"` | 小区（社区） |
| `issuesType: "3"` | 街区 |
| `issuesType: "4"` | 城区（城市） |

**禁止在回答中出现以下内容：**
- ❌ "类型1"、"类型2"、"类型3"、"类型4"
- ❌ "问题类型1"、"问题类型2"
- ❌ "住房（类型1）"、"小区（类型2）"、"街区（类型3）"
- ❌ 任何数字编码形式

**🚨 特别注意：不要在括号内显示编码！**
- ✅ "住房问题184条" 或 "住房：184条"
- ❌ "住房（类型1）问题184条" ← 这是错误的！
- ❌ "住房类（类型1）共184条" ← 这是错误的！
- 在表格或列表中，只写中文名称，不要加括号说明类型

**正确示例：**
- ✅ "住房问题共184条"
- ✅ "小区（社区）问题数量为73条"
- ✅ "街区问题有28条"

**错误示例（绝对禁止）：**
- ❌ "住房（类型1）问题184条"
- ❌ "住房类（类型1）共184条"
- ❌ "小区（类型2）73条"

## Required Output

- 最终只返回 Markdown 报告
- 不要返回 JSON
- 不要把最终结果包在 Markdown 代码块里
- 必须完整覆盖模板的一级、二级章节

## Workflow

1. 先通读用户提供的全部内容，识别并归入：
   - 城市问题
   - 更新建议
   - 立项依据
   - 项目建议
2. 再通过 `ReadSkillFile(skillId="cityagent-macro-update-report", relativePath="references/report-template.md")` 读取模板，严格按模板组织章节。
3. 如果输入同时包含四个维度信息，优先按“住房 / 小区（社区） / 街区 / 城区”组织；如果某维度信息缺失，保留章节结构并用保守表述补足。
4. 将输入改写为正式、完整、适合汇报材料的表述：
   - 保留用户已提供的具体指标、数量、地区、责任单位、投资额
   - 可对标题做概括提炼，但不得改写事实含义
   - 不照搬口语化或碎片化原文
5. 若任务中附带样例文本：
   - 只借鉴结构、语气、颗粒度
   - 不得借用样例中的地区、年份、数字、政策名称、项目名称
6. 当信息不足时：
   - 可用“国家、省、市相关政策提供了支持基础”之类的稳妥概括
   - 可用“资金安排可结合后续统筹渠道进一步研究落实”之类的保守表述
   - 不得擅自补具体政策编号、投资测算、责任单位或资金分担比例

## Hard Rules

- 只能基于本次输入内容生成报告
- 不得虚构输入中没有给出的核心事实数据
- 没有明确依据时，不要补造政策文件名、年份、具体金额、责任单位
- 风格保持正式、完整、可直接提交，避免宣传口号式语言

## 指标名称规范（重要）

- **必须使用完整指标名称**，如"存在结构安全隐患的住宅数量（栋）"、"中学服务半径覆盖率（%）"等
- **禁止使用指标代码**如"N-SSHU"、"N-GSHU"等，也**禁止使用"指标1xxx"、"维度1xxx"这类代号式称呼**
- 从原始输入或接口数据中获取的指标必须转译为完整名称
- 如果输入只有指标代码没有名称，必须先调用 `query_data(query="<指标代码>", sourceFiles="问题指标表.db")` 查询补全
- 报告正文中全程使用指标全称，不要混用代码和名称

## Writing Notes

- “城市问题”和“更新建议”优先用编号列表
- “立项依据”必须保留“政策支持”“资金支持”两个二级小节
- “项目建议”尽量覆盖项目目标、建设内容、实施主体、投资测算；没有信息时用保守表述补足结构
- 如输入信息只够支撑少量条目，也保留完整章节结构，不删主章节
