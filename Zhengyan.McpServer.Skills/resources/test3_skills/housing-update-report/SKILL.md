# ⚠️ 住房维度更新报告格式化 Skill（输出规范：不要出现"类型1"、"住房（类型1）"等字眼）

用于将住房维度更新内容整理成正式、完整、适合直接提交的 Markdown 报告。只适用于住房维度材料；如果任务是全市或多维度的城市体检宏观更新建议，应改用 `cityagent-macro-update-report`。

## Use When

- 用户明确要求把原始内容"整理成报告""改写成正式材料""输出 Markdown 报告"
- 输入内容与住房维度更新、住房问题、更新建议、立项依据、项目建议有关
- 需要把零散内容统一改写成固定章节结构
- 任务明确只针对住房维度，而不是"住房 + 小区 + 街区 + 城区"的城市级综合报告

## 问题类型编码（重要基础）

在调用需要问题类型参数的接口时，必须使用以下编码：

| 编码 | 问题类型 | 说明 |
|------|----------|------|
| 1 | 住房 | 住宅楼栋、住房单元等 |
| 2 | 小区（社区） | 小区、新村、花园、宿舍等 |
| 3 | 街区 | 街道、片区 |
| 4 | 城区（城市） | 城区、城市级 |

**重要：内部编码仅用于接口调用，在回答用户时必须转换为对应的中文名称！**

## 输出格式规范（重要！必须遵守）

当接口返回数据中包含 `target_type`、`issuesType` 等类型字段时，**必须**将编码转换为用户可读的中文名称：

| 接口返回 | 回答用户时必须说 |
|----------|-----------------|
| `issuesType: "1"` | 住房 |
| `issuesType: "2"` | 小区（社区） |
| `issuesType: "3"` | 街区 |
| `issuesType: "4"` | 城区（城市） |

**禁止在回答中出现以下内容：**
- ❌ "类型1"、"类型2"、"类型3"、"类型4"
- ❌ "问题类型1"、"问题类型2"
- ❌ "住房（类型1）"、"小区（类型2）"、"街区（类型3）"
- ❌ 任何数字编码形式

**🚨 特别注意：不要在括号内显示编码！**
- ✅ "住房问题184条" 或 "住房：184条"
- ❌ "住房（类型1）问题184条" ← 这是错误的！
- ❌ "住房类（类型1）共184条" ← 这是错误的！
- 在表格或列表中，只写中文名称，不要加括号说明类型

**正确示例：**
- ✅ "住房问题共184条"
- ✅ "小区（社区）问题数量为73条"
- ✅ "街区问题有28条"

**错误示例（绝对禁止）：**
- ❌ "住房（类型1）问题184条"
- ❌ "住房类（类型1）共184条"
- ❌ "小区（类型2）73条"

## Required Output

- 最终只返回 Markdown 报告
- 不要返回 JSON
- 不要把最终结果包在 Markdown 代码块里
- 必须完整覆盖报告模板中的所有一级和二级章节

## Workflow

1. 先通读用户提供的全部 `content`，识别可归入以下四类的信息：
   - 住房问题
   - 更新建议
   - 立项依据
   - 项目建议
2. 再通过 `ReadSkillFile(skillId="housing-update-report", relativePath="references/report-template.md")` 读取模板，严格按模板组织章节。
3. 将输入内容改写为正式、简洁、适合汇报材料的表述：
   - 保留用户已经提供的核心事实
   - 不照搬口语化、碎片化或清单式原文
   - 对标题进行概括提炼，但不要夸大
4. 若任务中提供了 `text.txt`：
   - 只借鉴其章节组织、语气和表达颗粒度
   - 不得复用其中的地名、数字、年份、政策名称、项目名称
5. 当信息不足时：
   - 允许使用稳妥概括
   - 不得擅自补充具体统计值、政策编号、项目投资额、责任单位名称
   - 可使用“待进一步明确”“可在后续论证中补充完善”这类保守表述

## Hard Rules

- 只能基于本次输入内容生成报告
- 不得虚构输入中没有提供的核心事实数据
- 如果任务是城市级、多维度、宏观更新建议，不要继续使用本 Skill，改走 `cityagent-macro-update-report`
- 若原始输入未给出具体政策名称，只能概括写"具备相关政策支持基础"之类的稳妥表述
- 若原始输入未给出明确资金来源，只能概括写"资金安排可结合后续统筹渠道进一步研究落实"
- 若原始输入未给出明确责任单位或投资测算，只能写为后续深化阶段进一步明确
- 风格保持正式、简洁、完整，避免口号式和宣传式语言

## 指标名称规范（重要）

- **必须使用完整指标名称**，如"存在结构安全隐患的住宅数量（栋）"、"中学服务半径覆盖率（%）"等
- **禁止使用指标代码**如"N-SSHU"、"N-GSHU"等，也**禁止使用"指标1xxx"、"维度1xxx"这类代号式称呼**
- 从原始输入中获取的指标必须转译为完整名称
- 如果输入只有指标代码没有名称，必须先调用 `query_data(query="<指标代码>", sourceFiles="问题指标表.db")` 查询补全
- 报告正文中全程使用指标全称，不要混用代码和名称

## Writing Notes

- “住房问题”与“更新建议”优先使用编号列表
- “立项依据”必须保留“政策支持”“资金支持”两个二级小节
- “项目建议”应尽量覆盖项目目标、建设内容、实施方向、责任分工、投资测算等；没有信息时用保守表述补足结构，但不要补造事实
- 如输入内容只够支撑 1 个问题或 1 条建议，也仍然保留完整章节结构，可将同类内容整合后表述为“重点问题”“重点建议”
